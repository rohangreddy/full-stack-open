Next.js and Route Handlers

The same ground the official material covers — Node, Express, REST, middleware — rebuilt on Next.js 16's App Router. Where the frameworks agree, the text barely changes. Where they don't, that's the part worth slowing down for.

Next.js and TypeScript

Was: “Node.js and Express”

In this part, our focus shifts towards the backend: implementing functionality on the server side of the stack. We'll build it on top of Next.js, a React framework that happens to include, as a first-class feature, a full server runtime — the same Node.js underneath, with routing, request handling, and JSON responses already wired in.

This means the relationship between "frontend framework" and "backend framework" collapses somewhat. In the original material, React (part 2) and Express (part 3) are two separate tools joined by fetch calls. Here, they're two capabilities of one framework: page.tsx files render UI, route.ts files answer HTTP requests, and both live in the same app/ directory.

This project was scaffolded with create-next-app and already has TypeScript, ESLint, and a dev server configured — there's no separate npm init step, and no index.js entry point to hand-write.

Version note This material assumes Next.js 16.2.10 (the version installed in this project's package.json). Next.js has a history of renaming and reshaping App Router conventions between major versions — check node_modules/next/dist/docs against whatever version you're actually running before trusting any specific API shown here.

Since this part isn't about the frontend, we won't be adding new pages built for humans to browse — just like the original material skips Vite for backend work, we'll mostly be adding files under app/api/ rather than new pages.

A route handler, not a web server

Was: “Simple web server”

The original tutorial's first step is to hand-build a web server with Node's built-in http module:

Express material · index.js
const http = require('http')

const app = http.createServer((request, response) => {
  response.writeHead(200, { 'Content-Type': 'text/plain' })
  response.end('Hello World')
})

const PORT = 3001
app.listen(PORT)
console.log(`Server running on port ${PORT}`)

Every one of those moving parts — createServer, writeHead, response.end, app.listen(PORT), the startup console.log — is infrastructure that Next.js's dev server already owns. There is no file in this project that plays the role of "the server." The closest unit of work is a Route Handler: a file named route.ts that exports a function per HTTP method.

Next.js · app/hello/route.ts
export async function GET() {
  return new Response('Hello World', {
    headers: { 'Content-Type': 'text/plain' },
  })
}

Run npm run dev and visit http://localhost:3000/hello — note the port. Next's dev server defaults to 3000, not 3001, so wherever the original text says :3001, read it as :3000 here. There's no EADDRINUSE ritual to learn either: if 3000 is taken, Next's CLI notices and offers the next free port itself.

One line from the original is worth flagging directly, because it's the one thing that doesn't survive the move:

Doesn't carry over “The server works the same way regardless of the latter part of the URL. Also the address http://localhost:3001/foo/bar will display the same content.” In Next.js, routing is filesystem-based — a route.ts answers only the exact path implied by its folder location. app/hello/route.ts handles /hello; a request to /hello/foo/bar 404s unless you deliberately add a catch-all segment for it.

The original also spends a paragraph on CommonJS vs. ES modules (require vs. import). That entire discussion is moot here: this is a TypeScript/ESM project end to end, so it's import/export everywhere, no require in sight.

Turning this into the JSON-notes version needs no new concept — just swap the string body for a serialized array, same as before:

Next.js · app/api/notes/route.ts
interface Note {
  id: string
  content: string
  important: boolean
}

const notes: Note[] = [
  { id: "1", content: "HTML is easy", important: true },
  { id: "2", content: "Browser can execute only JavaScript", important: false },
  { id: "3", content: "GET and POST are the most important methods of HTTP protocol", important: true },
]

export async function GET() {
  return Response.json(notes)
}

Response.json(notes) is doing exactly what JSON.stringify(notes) plus a manual Content-Type: application/json header did in the raw-http version — just in one call, with the header set for you. Visiting http://localhost:3000/api/notes shows the same formatted JSON the original screenshot shows.

Where Express's job already went

Was: “Express”

The original material's pitch for Express is that hand-rolling everything on raw http "is cumbersome, especially once the application grows in size" — so it reaches for a library that wraps the built-in module in a nicer interface: routing by path and verb, automatic JSON serialization, a request/response API that doesn't require manual header plumbing.

That entire pitch is the reason Route Handlers exist. There's no separate library to add here — npm install express has no equivalent step, because the convenience Express provides over raw http is exactly what Next.js's routing layer provides over raw http, just built into the framework you already installed. Nothing shows up in package.json's dependencies for this — it's not a dependency, it's the framework itself.

The rest of that section — semantic versioning, the caret in ^5.1.0, npm update, transitive dependencies in node_modules — is general npm knowledge, not Express-specific, and applies unchanged to a Next.js project's own dependencies (next, react, etc. in this project's package.json).

Web and Route Handlers

Was: “Web and Express”

The original defines two routes in one file — a human-facing root and a JSON API path:

Express material
const express = require('express')
const app = express()

app.get('/', (request, response) => {
  response.send('<h1>Hello World!</h1>')
})

app.get('/api/notes', (request, response) => {
  response.json(notes)
})

Express routes are registered imperatively — you call app.get(path, handler) for each one, in one file, in whatever order you write them. Next.js routes are declared by file location — there's no registry to call into; the path a handler answers to is the folder it lives in.

That difference matters most for the first route. app.get('/', ...) sends an HTML string — and the literal equivalent of "root path returns HTML" in this project isn't a Route Handler at all, it's app/page.tsx, which already occupies / and already returns markup (as JSX, which Next renders to HTML). A route.ts can't even coexist with a page.tsx at the same segment — so for a genuine "page" at /, page.tsx is the tool, not route.ts.

The second route needs nothing new — it's the app/api/notes/route.ts file from the previous section, doing precisely what response.json(notes) did: serialize notes, set Content-Type: application/json, default to status 200.

Express
app.get('/api/notes', (req, res) => {
  res.json(notes)
})
Next.js
export async function GET() {
  return Response.json(notes)
}

Same guarantee the original calls out — no manual JSON.stringify needed on either side, both frameworks do the string transformation for you and set the header to match.

Automatic change tracking

Was: “Automatic Change Tracking”

The original spends this whole section getting from "restart the server by hand after every edit" to a workable node --watch index.js, then wires it into a custom "dev" npm script so npm run dev restarts the server on save.

Already done for you create-next-app already generated a "dev": "next dev" script in this project's package.json. There's no --watch flag to discover and no script to author — file-watching and process reloading are default behavior of next dev, on day one.

One caveat from the original does still apply, almost word for word: editing a Route Handler doesn't get the same live-patching treatment React components get in the browser. Next's Fast Refresh story is about UI trees, not JSON responses — so after changing route.ts logic, you still refresh the browser (or re-send the request) to see the new output, even though the server process itself picked up the change automatically.

REST

This section barely changes at all — REST is a description of an HTTP contract, not a framework feature, so it applies identically regardless of what's implementing the server side. A note resource with id 10 still lives at a URL like www.example.com/api/notes/10; the collection is still www.example.com/api/notes; the verb-to-operation mapping is unchanged:

URLVerbFunctionality
notes/10GETfetches a single resource
notesGETfetches all resources in the collection
notesPOSTcreates a new resource based on the request data
notes/10DELETEremoves the identified resource
notes/10PUTreplaces the entire identified resource with the request data
notes/10PATCHreplaces a part of the identified resource with the request data

Whether the code answering these requests is an Express route or a Next.js route.ts export is invisible from the client's side of the wire — a REST client, browser, or Postman can't tell the difference, and shouldn't need to.

Fetching a single resource

Express reads the id out of the URL with colon-syntax route params:

Express
app.get('/api/notes/:id', (request, response) => {
  const id = request.params.id
  const note = notes.find(note => note.id === id)
  response.json(note)
})

Next.js's equivalent of a route param is a dynamic segment — a folder named in square brackets. The id shows up as a promise-wrapped param your handler must await, not a synchronous property:

Next.js · app/api/notes/[id]/route.ts
export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params
  const note = notes.find(note => note.id === id)
  return Response.json(note)
}
Why a promise? This is a genuine Next.js-specific wrinkle, not a stylistic choice: params became a Promise as of Next 15, to support cases where route params aren't known until request time even for otherwise-static routes. Forgetting the await is a common mistake — TypeScript will catch it for you here, since params.id on a Promise is a type error.

The same problem the original walks into next — an id that doesn't match anything resolves to undefined, and blindly returning it produces a 200 with an empty body, which is the wrong status for "not found" — applies exactly the same way here, and the fix is the same shape:

Express
if (note) {
  response.json(note)
} else {
  response.status(404).end()
}
Next.js
if (note) {
  return Response.json(note)
}
return new Response(null, { status: 404 })

Same reasoning as the original: all objects are truthy, undefined is falsy, and a REST API doesn't need to render anything human-readable for a 404 — the status code alone is the interface.

Deleting resources

Same shape as the GET-by-id case, different exported function name and verb:

Express
app.delete('/api/notes/:id', (request, response) => {
  const id = request.params.id
  notes = notes.filter(note => note.id !== id)
  response.status(204).end()
})
Next.js
export async function DELETE(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params
  notes = notes.filter(note => note.id !== id)
  return new Response(null, { status: 204 })
}

Both files live at the same path — app/api/notes/[id]/route.ts exports both GET and DELETE side by side, the same way one Express file has both an app.get('/api/notes/:id', ...) and an app.delete('/api/notes/:id', ...). The same unresolved question from the original applies unchanged too: no consensus on 204 vs. 404 when the id never existed in the first place — 204 both times is the pragmatic default.

One TypeScript wrinkle let notes = [...] needs to be a mutable let, not const, for reassignment via .filter() to work — same as the original Express example, this is plain JavaScript/TypeScript behavior, nothing Next-specific about it.

Postman & REST clients

Nothing here is framework-specific, and none of it changes: Postman, the VS Code REST Client, and the WebStorm HTTP Client all talk raw HTTP, so they work identically against a Next.js Route Handler as they do against an Express route. The only adjustment is the port in every URL — http://localhost:3000/api/notes/2, not :3001, since that's what next dev listens on by default.

A .rest file for this project looks exactly like the original:

GET http://localhost:3000/api/notes/

###
POST http://localhost:3000/api/notes/ HTTP/1.1
content-type: application/json

{
    "content": "sample",
    "important": false
}

Receiving data

Here's a place Next.js is strictly simpler than Express, not just relabeled. The original needs a dedicated step to make request.body exist at all:

Express
const express = require('express')
const app = express()

app.use(express.json())

app.post('/api/notes', (request, response) => {
  const note = request.body
  console.log(note)
  response.json(note)
})

Without express.json() registered as middleware, request.body is undefined — the json-parser middleware is what reads the raw request stream, parses it, and attaches the result to request.body before your handler runs.

Next.js's Request object is the Web-standard Request — body parsing is just a method call on it, no separate parser to install or register:

Next.js
export async function POST(request: Request) {
  const note = await request.json()
  console.log(note)
  return Response.json(note)
}

There's no app.use(express.json()) line anywhere — no middleware to register before routes, no ordering to get right relative to a logger. await request.json() reads and parses the body on demand, right where you need it. The failure mode the original warns about — a client sending the wrong Content-Type (e.g. text/plain) and the server silently receiving an empty object — still exists here in a similar shape: request.json() will throw on a body that isn't valid JSON, so a request with the wrong content type or a malformed payload surfaces as a thrown error you'll want to catch, rather than a silently empty object.

The id-generation and validation logic translates over almost line for line, with one difference forced by TypeScript: the body from request.json() is typed any, so reading its shape safely means either casting it or checking it defensively before trusting it:

Next.js · app/api/notes/route.ts
const generateId = (): string => {
  const maxId = notes.length > 0
    ? Math.max(...notes.map(n => Number(n.id)))
    : 0
  return String(maxId + 1)
}

export async function POST(request: Request) {
  const body = await request.json()

  if (!body.content) {
    return Response.json({ error: 'content missing' }, { status: 400 })
  }

  const note: Note = {
    content: body.content,
    important: body.important || false,
    id: generateId(),
  }

  notes = notes.concat(note)

  return Response.json(note)
}

The reasoning is identical to the original at every step: returning early on the 400 branch matters for the same reason — skip it, and a note with no content gets saved before the function reaches its end; body.important || false defaults a missing/falsy important to false the same way in both languages, since the || operator behaves identically in TypeScript and JavaScript.

HTTP request types

This section is entirely about the HTTP standard itself — safety and idempotency are properties of the protocol, not of whatever framework happens to be implementing a given endpoint, so nothing here changes when the implementation is a Next.js Route Handler instead of an Express route.

GET and HEAD are still expected to be safe — no side effects, nothing but retrieval. GET, HEAD, PUT, and DELETE are still expected to be idempotent — N identical requests should leave the server in the same state one would. POST is still the one verb that's neither: five POSTs with the same body to app/api/notes/route.ts's POST handler produce five distinct notes, same as five POSTs to an Express route would.

None of these guarantees come from the framework in either case — they come from writing handlers that respect the convention, whether that handler is registered with app.get(...) or exported as GET from a route.ts file.

Where middleware went

Was: “Middleware” — the biggest structural difference in this whole part

This is the one section where the two frameworks genuinely part ways, so it's worth being precise about what's actually gone versus what's just relocated.

Express's middleware model is a chain: every registered function receives (request, response, next), and calling next() passes control to whatever's registered after it. A request logger looks like this:

Express
const requestLogger = (request, response, next) => {
  console.log('Method:', request.method)
  console.log('Path:  ', request.path)
  console.log('Body:  ', request.body)
  console.log('---')
  next()
}

app.use(requestLogger)

Route Handlers don't have a next()-chaining mechanism at all — there's no per-route middleware pipeline to register logging or parsing into. That leaves two real options, and they solve different problems:

Option A — log inline, in the handler itself

For a handful of routes, the simplest port of the original's intent is a small helper function you call explicitly at the top of each handler:

function logRequest(request: Request) {
  console.log('Method:', request.method)
  console.log('Path:  ', new URL(request.url).pathname)
}

export async function GET(request: Request) {
  logRequest(request)
  return Response.json(notes)
}

This is more repetitive than Express's one-line app.use(requestLogger) — you're opting every handler in individually rather than registering once — but it stays local to each route and needs no new file-convention knowledge.

Option B — proxy.ts, for logic that spans every route

Naming, precisely In Next.js 15 and earlier this file was called middleware.ts. As of Next.js 16 — the version this project runs — it was renamed to proxy.ts "to better reflect its purpose"; the functionality is unchanged. If you're reading older Next.js material or Stack Overflow answers, mentally substitute proxy.ts for middleware.ts.

A proxy.ts at the project root runs before a request reaches any route, and can inspect or modify it. It's the closest thing to Express's "runs before all routes" middleware — but it's a single file per project (you compose logic into it via imports, not via a growing app.use() stack), and it's tuned for redirects, rewrites, and header manipulation rather than general-purpose request logging:

// proxy.ts
import type { NextRequest } from 'next/server'

export function proxy(request: NextRequest) {
  console.log('Method:', request.method)
  console.log('Path:  ', request.nextUrl.pathname)
}

export const config = {
  matcher: '/api/:path*',
}

The matcher config is doing the job the original's middleware ordering did implicitly — scoping which paths this runs for, rather than relying on the order functions were registered in.

The 404 case needs nothing at all

The original's last piece of middleware catches requests to routes that don't exist:

Express
const unknownEndpoint = (request, response) => {
  response.status(404).send({ error: 'unknown endpoint' })
}

app.use(unknownEndpoint)
Already the default Next.js returns a 404 automatically for any request that doesn't match a route.ts, page.tsx, or other file convention — there's no catch-all middleware to write. If you want a custom JSON body on that 404 rather than Next's default page, the tool is a not-found.tsx file (for page routes) — there isn't a direct equivalent for shaping the body of an unmatched API route's 404 the way unknownEndpoint does, since there's no handler being invoked at all in that case.

Exercises 3.7–3.8

Logging, without morgan

3.7Phonebook backend step 7

The original installs morgan and wires it in with app.use(morgan('tiny')). Morgan is an Express-middleware library specifically — it hooks into the (req, res, next) chain, which Route Handlers don't have, so there's no drop-in equivalent package to install here.

Port the intent, not the library: write a small logging helper and call it at the top of each handler you want logged (Option A above), mirroring tiny's format — method, path, status, response time:

function logTiny(method: string, path: string, status: number, startedAt: number) {
  const ms = Date.now() - startedAt
  console.log(`${method} ${path} ${status} - ${ms}ms`)
}

3.8*Phonebook backend step 8

The original's optional challenge is configuring morgan to also log the POST body — here, that's just extending the same helper to accept and print the parsed body from await request.json(), same privacy caveat as the original: think twice before logging real request bodies anywhere persistent, since names and phone numbers are exactly the kind of data GDPR and similar laws care about.
