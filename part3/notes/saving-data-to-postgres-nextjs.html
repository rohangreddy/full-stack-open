Saving Data to Postgres

The original material reaches for MongoDB and Mongoose. This version covers the same ground — a real, persistent, cloud-hosted database wired into the backend — using a relational database (Postgres) and the ORM that's become the default pairing with Next.js: Prisma.

Debugging Next.js applications

Was: “Debugging Node applications”

Debugging a Next.js Route Handler is slightly more involved than debugging a client component, but the fundamentals are unchanged from the original material: printing to the console is a tried and true method, and it's always worth doing first. Route Handler code runs on the server, so its console.log output shows up in the terminal running next dev, not the browser console — that trips people up the first time.

Visual Studio Code

The VS Code debugger works the same way it does for any Node process — you just point it at the next dev script instead of a hand-written index.js. A minimal .vscode/launch.json entry looks like this:

{
  "type": "node",
  "request": "launch",
  "name": "Next.js: debug server-side",
  "runtimeExecutable": "npm",
  "runtimeArgs": ["run", "dev"],
  "skipFiles": ["<node_internals>/**"]
}

Set a breakpoint inside a route.ts handler, launch that configuration, and trigger the route from the browser or a REST client — execution pauses exactly like it would in the original Express example, with the same variable inspector and step controls.

Client vs. server code Next.js blurs frontend and backend into one project, so it matters which side you're debugging. A breakpoint in app/page.tsx is client code — use the browser's own dev tools for that, same as any React app. A breakpoint in app/api/persons/route.ts is server code — that's what the VS Code launch config above, or node --inspect-style debugging, is for.

Chrome dev tools

Next.js's dev server exposes the same --inspect hook the original material uses. Start it with:

NODE_OPTIONS='--inspect' npm run dev

Then open chrome://inspect in Chrome (rather than clicking a green Node icon from a separate dev-tools panel, which was specific to how the original example launched its server) and connect to the remote target. From there, the debugging view works exactly as described in the original: the Sources tab sets breakpoints in server code, the Console tab shows every console.log the server produces for requests you trigger, and you can inspect and evaluate expressions live.

All of a route handler's log lines land in this same server console — including anything logged from the database layer we're about to add — so this is also where Prisma's own query logging will show up, if you enable it.

Question everything

This advice is unchanged by the choice of framework or database, and is arguably more relevant here than ever: once a database is in the mix, there are now three layers where a bug can hide — the React tree, the Route Handler, and the database itself — and it's very common for the actual problem to be one layer away from where it first appears to be.

The discipline is the same as the original material describes: be systematic, question everything, and eliminate possibilities one at a time using logging, a REST client, the debugger, and experience. When something breaks, the worst response is to keep writing new code on top of the broken state — stop, and either fix the root cause or delete back to a point where things worked, exactly as the Toyota "stop and fix" (Jidoka) principle the original text cites.

Postgres, not MongoDB

Was: “MongoDB”

The official material reaches for MongoDB — a document database — specifically because it's lower complexity than a relational database for a first pass at persistence, deferring relational databases and SQL to part 13 of the course.

This version makes the opposite trade deliberately: Postgres, a relational database, paired with Prisma, an ORM that has become the default choice in the Next.js ecosystem specifically because it integrates cleanly with TypeScript and the App Router's server-only code. Where the original spends time explaining what a "document" and a "collection" are, the equivalent concepts here are the familiar relational ones — a table made of rows, each with a fixed set of typed columns — which likely need less introduction if you've touched SQL before.

Why Prisma specifically You could talk to Postgres directly with a lower-level client library (e.g. pg or postgres.js), the same way the original material could've used the plain MongoDB Node.js driver directly. Prisma occupies the same niche Mongoose does in the original text: a higher-level layer that turns database rows into typed objects, and gives you migrations and a schema file as the source of truth for your data's shape.

One structural difference is worth flagging up front, because it shapes several sections below: Mongo is schemaless at the database level — Mongoose is what imposes structure, at the application layer. Postgres is not schemaless — the database itself enforces column types and constraints, and Prisma's schema file is used to both describe that structure and generate the actual SQL that creates it (via migrations). There's no equivalent step in the Mongoose workflow, where the schema is purely an application-level convention the database itself doesn't know about.

Creating a cloud Postgres database

Was: “MongoDB Atlas” setup

Just as the original material points you at MongoDB Atlas for a free, cloud-hosted cluster, you'll want a cloud-hosted Postgres instance rather than installing Postgres locally. Neon is a common pick for Next.js projects — free tier, serverless-friendly, and it hands you a connection string in the same spirit as Atlas's. (Supabase and Vercel Postgres are equally reasonable alternatives; the workflow below is generic to any of them.)

The steps mirror the Atlas flow closely:

  1. Create an account and a new project. A default database and a default Postgres role (Neon's term for a database user) are usually created for you automatically — this is the rough equivalent of the "create a cluster" step for Atlas.
  2. Unlike Atlas, there's typically no separate "network access / IP allowlist" step to configure for a serverless Postgres provider — connections are authenticated by the credentials in the connection string itself, not by source IP. Still, double-check your provider's dashboard for any access restriction it applies by default.
  3. Copy the connection string from the dashboard's "Connect" panel. It has the same job as the Atlas URI, just a different shape:
postgresql://fullstack:thepasswordishere@ep-example-12345.eu-central-1.aws.neon.tech/neondb?sslmode=require
Same warning as the original This connection string embeds a password in plain text — treat it exactly as the original material treats the MongoDB URI: never commit it, never paste it into a public issue or chat. We'll keep it out of version control the same way, just via a different (and simpler) mechanism than the original's dotenv setup — see Environment variables, the Next.js way below.

We are now ready to use the database.

Prisma: schema and client

Was: “Mongoose”

Install Prisma in the phonebook project:

npm install prisma --save-dev
npm install @prisma/client
npx prisma init

npx prisma init is the step with no real Mongoose equivalent — the original material's mongoose.Schema is just a JavaScript object literal you write by hand anywhere in your code. Prisma instead scaffolds a dedicated prisma/schema.prisma file (Prisma's own schema language, not TypeScript) and a .env file with a DATABASE_URL placeholder already wired in.

Paste the Neon connection string from the previous section into that generated .env as DATABASE_URL, then define the schema for a phonebook entry:

prisma/schema.prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model Person {
  id     Int    @id @default(autoincrement())
  name   String
  number String
}

This plays the same role noteSchema plays in the original — it's the single source of truth for what a stored record looks like — but with one addition the document-database version doesn't need: datasource db, which is where DATABASE_URL is actually consumed. In the original, the connection URL is passed directly to mongoose.connect(url) in application code; here, it's read once, by Prisma's tooling, from the schema file.

Unlike Mongoose — where defining a schema is purely an in-application convention the database itself never sees — this schema is turned into real SQL and applied to the database with a migration:

npx prisma migrate dev --name init

This is a genuinely new step with no equivalent in the original material: it creates the Person table in Postgres (Mongo, being schemaless, never needed a table/collection created up front — MongoDB Atlas creates a collection automatically the first time a document is written to it) and generates a fully-typed Prisma Client tailored to this exact schema.

Naming convention, inverted The original notes that Mongoose pluralizes a singular model name into the collection name (Note model → notes collection, Person model → people collection). Prisma Client does something similar in spirit but keeps the casing convention instead of pluralizing awkwardly: a model Person is queried through prisma.person (camelCased, not pluralized to people) — worth checking directly against the generated client rather than assuming either convention.

Exercise 3.12

3.12Command-line database

Create a cloud-based Postgres database for the phonebook application (Neon, Supabase, or any provider that gives you a DATABASE_URL-style connection string).

Create a prisma/schema.prisma with a Person model as shown above, and run npx prisma migrate dev --name init to create the table.

Write a small script — scripts/db-cli.ts, run with npx tsx scripts/db-cli.ts — that can be used for adding entries to the phonebook and listing all existing entries, mirroring the original exercise's mongo.js in spirit:

import { PrismaClient } from '../generated/prisma'

const prisma = new PrismaClient()

const [name, number] = process.argv.slice(2)

async function main() {
  if (name && number) {
    const person = await prisma.person.create({ data: { name, number } })
    console.log(`added ${person.name} number ${person.number} to phonebook`)
  } else {
    const people = await prisma.person.findMany()
    console.log('phonebook:')
    people.forEach(p => console.log(`${p.name} ${p.number}`))
  }
}

main().finally(() => prisma.$disconnect())

Usage mirrors the original exercise, minus the password argument — the connection string already lives in .env, so there's nothing sensitive to pass on the command line at all:

npx tsx scripts/db-cli.ts "Arto Vihavainen" 045-1232456
npx tsx scripts/db-cli.ts
Still true here The original's warning not to commit the password still applies — just to a different file. npx prisma init already added prisma/.env-style files to .gitignore for you (Next.js's own .env* gitignore rule covers the project-root .env too) — double check before your first commit rather than assuming.

Connecting the backend to a database

With the schema migrated, we have enough to start using Postgres in the phonebook's route handlers. The direct, quick-and-dirty version — mirroring the original's first pass of pasting Mongoose setup straight into index.js — would be creating a PrismaClient right inside a route file:

app/api/persons/route.ts (quick pass, not final)
import { PrismaClient } from '../../../generated/prisma'

const prisma = new PrismaClient()

export async function GET() {
  const people = await prisma.person.findMany()
  return Response.json(people)
}

This already fixes something the original material calls out about Mongoose's default output: Mongo documents carry a _id object and a __v version key that need a toJSON transform to clean up before sending them to the frontend. A Prisma Person row has exactly the fields defined in the schema — id, name, number — serialized as-is by Response.json, no transform step needed. The one adjustment on the frontend side: id here is a Postgres Int, not a Mongo ObjectId string, so anywhere the frontend was treating ids as opaque strings still works, but anywhere it was relying on Mongo's specific 24-character hex format will need to change.

The problem with the snippet above is the one the next section exists to fix — exactly as the original material moves its inline Mongoose setup into models/note.js.

The Prisma client as its own module

Was: “Moving db configuration to its own module”

The original extracts Mongoose setup into models/note.js mainly for organization — nothing about Express forces a single shared connection object. With Prisma in a framework that hot-reloads server code on every save, this step stops being optional: instantiating new PrismaClient() inside a route file (as above) creates a fresh client, and a fresh pool of database connections, on every Fast-Refresh reload in development — which reliably exhausts your database's connection limit. This is a documented Next.js + Prisma pitfall, not a stylistic preference.

The fix is a small singleton module, conventionally placed at lib/prisma.ts (matching this project's existing lib/persons.ts convention):

lib/prisma.ts
import { PrismaClient } from '../generated/prisma'

const globalForPrisma = globalThis as unknown as {
  prisma: PrismaClient | undefined
}

export const prisma = globalForPrisma.prisma ?? new PrismaClient()

if (process.env.NODE_ENV !== 'production') {
  globalForPrisma.prisma = prisma
}

The intent maps directly onto the original's module: define the connection once, export the ready-to-use client, and let every route handler import the same instance rather than each managing its own. The mechanism differs because the failure mode differs — Mongoose's models/note.js is a plain module-caching trick (Node only evaluates a module once per process); Prisma's version stashes the client on globalThis specifically to survive Next.js's dev-time module reloading, which a plain module-level const would not.

Importing it elsewhere reads exactly like the original's const Note = require('./models/note'):

import { prisma } from '../../../lib/prisma'

Environment variables, the Next.js way

Was: “Defining environment variables using the dotenv library”

The original material installs a separate library, dotenv, and adds an explicit require('dotenv').config() call — with a specific warning that it must run before any module that reads process.env is imported, since import order determines whether the variables exist yet.

Already the default — nothing to install Next.js loads .env, .env.local, and friends automatically, with no library and no explicit config() call, for any file under app/ (Route Handlers included) as well as for the Prisma CLI itself. There is no import-order hazard to manage — the values are already in process.env before your code runs, full stop.

npx prisma init already created a root .env with DATABASE_URL in it, which is both what Prisma's CLI reads for migrations and what your route handlers read at request time via process.env.DATABASE_URL — one file, one variable, used by both. This project's .gitignore (generated by create-next-app) already excludes .env*, so — unlike the original material, where gitignoring .env is a step you have to remember to do yourself — there's nothing extra to configure here.

The original's PORT environment variable has no equivalent worth adding: next dev and next start both default to port 3000 and take a -p flag directly if you need to change it, so there's no app.listen(PORT) call in this codebase for an env var to feed into.

The original's platform-specific notes about Fly.io's .dockerignore and Render's dashboard env panel carry over conceptually to whatever you deploy this to — Vercel, Render, Fly.io, etc. all have a "set an environment variable in the dashboard" step for production secrets like DATABASE_URL, since a .env file that's gitignored locally obviously isn't available on a server that only receives what you push or deploy.

Using the database in route handlers

With the shared client in place, the GET handler becomes:

app/api/persons/route.ts
import { prisma } from '../../../lib/prisma'

export async function GET() {
  const people = await prisma.person.findMany()
  return Response.json(people)
}

Where the original awaits a .then() callback off Note.find({}), Prisma Client methods are promise-based the same way — async/await reads slightly more directly here only because Route Handlers are already async functions by convention, not because of anything Prisma-specific.

Creating a new entry:

app/api/persons/route.ts
export async function POST(request: Request) {
  const body = await request.json()

  if (!body.name || !body.number) {
    return Response.json({ error: 'name or number missing' }, { status: 400 })
  }

  const person = await prisma.person.create({
    data: { name: body.name, number: body.number },
  })

  return Response.json(person)
}

Same shape as the original's note.save().then(savedNote => response.json(savedNote)): validate first and return early on failure, then persist, then respond with exactly what got saved (including the database-assigned id) rather than echoing back the raw request body.

Fetching a single record by id, the Prisma equivalent of Note.findById, is findUnique — and because the schema's id is an Int, the string route param needs an explicit conversion Mongoose never required (Mongo ObjectIds are handled as opaque strings throughout):

app/api/persons/[id]/route.ts
import { prisma } from '../../../../lib/prisma'

export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params
  const person = await prisma.person.findUnique({
    where: { id: Number(id) },
  })

  if (person) {
    return Response.json(person)
  }
  return new Response(null, { status: 404 })
}

Same 404-on-null branching the original establishes for findById — a nonexistent id resolves to null here too, not an error, so the check is a plain truthiness test either way.

Error handling

Requesting a note with a nonexistent-but-otherwise-valid id resolves to null in the original — handled with a plain if (note) check, already covered above for the Prisma equivalent. The more interesting case is the original's second scenario: an id that isn't even the right shape for the database's id type, e.g. /api/notes/someInvalidId against Mongo's ObjectId format, which makes Mongoose's findById throw a CastError that has to be caught explicitly.

The same class of problem exists here: our schema's id is an Int, so a request to /api/persons/someInvalidId needs to be handled before it ever reaches Prisma, since Number('someInvalidId') is NaN, not a thrown error — a plain wrong-format id fails silently rather than loudly, which is arguably worse than Mongoose's approach if left unchecked:

app/api/persons/[id]/route.ts
export async function GET(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params
  const numericId = Number(id)

  if (!Number.isInteger(numericId)) {
    return Response.json({ error: 'malformatted id' }, { status: 400 })
  }

  const person = await prisma.person.findUnique({ where: { id: numericId } })

  if (person) {
    return Response.json(person)
  }
  return new Response(null, { status: 404 })
}

Prisma does still throw its own runtime errors — a unique-constraint violation on create(), for instance, throws a PrismaClientKnownRequestError tagged with a code like P2002 (unique constraint failed) or P2025 (record not found on an update/delete). Those are worth catching explicitly the same way the original catches a rejected Mongoose promise:

import { Prisma } from '../../../../generated/prisma'

try {
  const person = await prisma.person.update({
    where: { id: numericId },
    data: { number: body.number },
  })
  return Response.json(person)
} catch (error) {
  if (
    error instanceof Prisma.PrismaClientKnownRequestError &&
    error.code === 'P2025'
  ) {
    return new Response(null, { status: 404 })
  }
  throw error
}
No middleware chain to centralize this in The original moves this logic into a single Express errorHandler middleware, registered once with app.use(errorHandler) at the end of the chain, so every route benefits without repeating the catch block. Route Handlers have no equivalent "runs after every route" mechanism (see the proxy.ts discussion in the previous chapter of these notes) — the practical option is a small shared helper function you call from each handler's catch block, trading Express's one-line centralization for a few extra explicit calls.

Other operations

Deleting an entry, the Prisma equivalent of findByIdAndDelete:

app/api/persons/[id]/route.ts
export async function DELETE(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params

  try {
    await prisma.person.delete({ where: { id: Number(id) } })
    return new Response(null, { status: 204 })
  } catch (error) {
    if (
      error instanceof Prisma.PrismaClientKnownRequestError &&
      error.code === 'P2025'
    ) {
      return new Response(null, { status: 204 })
    }
    throw error
  }
}

Note the deliberate choice to still return 204 when the row was already gone: Prisma's delete(), unlike Mongoose's findByIdAndDelete, throws (P2025) rather than silently succeeding when nothing matched the id — so without the catch here, "delete something that's already deleted" would surface as a 500 instead of the idempotent 204 the original settles on for both cases.

Updating an entry — the original walks through findById then mutate-then-save() specifically to preserve Mongoose's validation, favoring that over the shortcut findByIdAndUpdate. Prisma's update() doesn't have that same validation trade-off (schema-level constraints are enforced by Postgres itself either way), so the direct one-call version is the idiomatic choice here, not a shortcut to be wary of:

app/api/persons/[id]/route.ts
export async function PUT(
  request: Request,
  { params }: { params: Promise<{ id: string }> }
) {
  const { id } = await params
  const body = await request.json()

  try {
    const person = await prisma.person.update({
      where: { id: Number(id) },
      data: { name: body.name, number: body.number },
    })
    return Response.json(person)
  } catch (error) {
    if (
      error instanceof Prisma.PrismaClientKnownRequestError &&
      error.code === 'P2025'
    ) {
      return new Response(null, { status: 404 })
    }
    throw error
  }
}

No nested-promise concern here either — the original flags its update handler's nested .then() inside .then() as something async/await (introduced later in the course) will eventually clean up. Since this project is already using async/await throughout, that awkwardness never shows up in the first place.

When the frontend tries to add a person whose name is already in the phonebook, it sends a PUT to that person's URL to update their number instead. Support that in the PUT handler shown above (it already does), and additionally add a @unique constraint on the name field in schema.prisma — a genuinely relational-database way to enforce the invariant at the data layer rather than only in application code:

model Person {
  id     Int    @id @default(autoincrement())
  name   String @unique
  number String
}

Run npx prisma migrate dev again after changing the schema. With the constraint in place, a duplicate create() throws P2002, which your POST handler should catch and turn into a 400 with a body like { error: 'name must be unique' } — mirroring the original exercise's application-level uniqueness check, but backed by an actual database constraint instead of a manual .find() scan.
