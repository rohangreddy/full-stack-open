Converting the Notes Frontend to Next.js

A step-by-step port of part2/src (Vite + React + axios) into the Next.js/TypeScript project from earlier in this part. Along the way we clean up a few things the original code got away with only because it was a learning exercise, not because they were good practice.

Starting point

The code we're converting is the notes frontend built up over part 2: four files, no TypeScript, axios for HTTP, and a component tree that fetches its own data in a useEffect after the first render.

part2/src/services/notes.js
import axios from 'axios'
const baseUrl = 'http://localhost:3001/notes'

const getAll = () => {
  const request = axios.get(baseUrl)
  return request.then(response => response.data)
}

const create = newObject => {
  const request = axios.post(baseUrl, newObject)
  return request.then(response => response.data)
}

const update = (id, newObject) => {
  const request = axios.patch(`${baseUrl}/${id}`, newObject)
  return request.then(response => response.data)
}

export default { getAll, create, update }
part2/src/components/Note.jsx
const Note = ({ note, toggleImportance }) => {
  const label = note.important
    ? 'make not important' : 'make important'

  return (
    <li>
      {note.content}
      <button onClick={toggleImportance}>{label}</button>
    </li>
  )
}

export default Note

App.jsx is the biggest file: it owns notes, newNote, and showAll state, fetches notes in a useEffect on mount, and defines toggleImportanceOf and addNote handlers that call into noteService. Nothing about that shape is wrong for a Vite SPA — it's exactly what the course teaches at this point. Some of it stops being appropriate once the same code moves into a framework with a server half, which is what most of this walkthrough is about.

What's worth fixing along the way

Before touching any files, it's worth naming the things in the original code that a straight line-for-line port would happily carry forward unchanged — some are Next.js-specific opportunities, others are just rough edges from a learning exercise:

IssueWhereWhy it's worth fixing here

None of these are exotic fixes — they're the same review comments this code would get in any TypeScript codebase. The Next.js migration is just a natural point to make them, since several files are being rewritten anyway.

1Give the domain a type

Nothing in the original code declares what a note actually looks like — it's implicit in how the object is used across four files. One Note interface, defined once, replaces that implicit contract:

lib/notes.ts (top of file)
export interface Note {
  id: string
  content: string
  important: boolean
}

Every file below imports this instead of re-describing the shape. If the backend's shape ever changes, TypeScript will now point at every call site that needs updating instead of leaving it to a runtime surprise.

2Fetch the initial notes on the server

The original loads notes like this:

part2/src/App.jsx (excerpt)
const [notes, setNotes] = useState([])

const hook = () => {
  noteService
    .getAll()
    .then(initialNotes => {
      setNotes(initialNotes)
    })
}

useEffect(hook, [])

This is the textbook shape for "fetch data when a component mounts" in a client-only React app, and it's exactly what the You Might Not Need an Effect guide flags as unnecessary once there's somewhere else for the fetch to happen. In a Next.js project there is: app/page.tsx is a Server Component by default, which means it can simply await data before returning any JSX at all — no empty initial render, no effect, no useState just to hold data that's known before the page ever reaches the browser.

Don't fetch your own API over HTTP from a Server Component The tempting move is await fetch('/api/notes') straight from page.tsx — but Server Components have no implicit origin the way a browser tab does, so a relative URL doesn't resolve, and an absolute one means a real network round-trip to yourself for data your own process already has in memory. The better shape: pull the data-access code the route handler uses into a shared module, and have both the route handler and the page import it directly.

That means one small refactor to the backend built earlier in this part — move the in-memory notes array and its helpers out of app/api/notes/route.ts and into lib/notes-store.ts, then have the route handler import from there instead of declaring it inline:

lib/notes-store.ts
import type { Note } from './notes'

export let notes: Note[] = [
  { id: '1', content: 'HTML is easy', important: true },
  { id: '2', content: 'Browser can execute only JavaScript', important: false },
  { id: '3', content: 'GET and POST are the most important methods of HTTP protocol', important: true },
]

export const getAll = async (): Promise<Note[]> => notes

With that in place, page.tsx becomes an async function that awaits the data directly — the same pattern the Next.js data-fetching guide recommends for Server Components reading from a database or ORM, just applied to an in-memory store here:

app/page.tsx
import { getAll } from '../lib/notes-store'
import NotesApp from './components/NotesApp'

export default async function Page() {
  const initialNotes = await getAll()
  return <NotesApp initialNotes={initialNotes} />
}

The notes variable is exported as let, not const, for the same reason as in the earlier Route Handlers chapter of these notes: the DELETE/POST handlers reassign it via .filter()/.concat().

3Draw the server/client boundary

Everything past the initial fetch in the original — the newNote input, the showAll toggle, toggleImportanceOf, addNote — is genuinely interactive: state, event handlers, a controlled input. That's precisely the "when you need interactivity" criterion Next.js uses to decide something belongs in a Client Component, so it becomes one file, marked with the 'use client' directive, that receives the server-fetched notes as a prop rather than fetching them itself:

app/components/NotesApp.tsx
'use client'

import { useState } from 'react'
import Note from './Note'
import { createNote, updateNote } from '../../lib/notes'
import type { Note as NoteType } from '../../lib/notes'

export default function NotesApp({ initialNotes }: { initialNotes: NoteType[] }) {
  const [notes, setNotes] = useState(initialNotes)
  const [newNote, setNewNote] = useState('')
  const [showAll, setShowAll] = useState(true)

  const toggleImportanceOf = (id: string) => {
    const note = notes.find(n => n.id === id)
    if (!note) return
    const changedNote = { ...note, important: !note.important }

    updateNote(id, changedNote)
      .then(updatedNote => {
        setNotes(notes.map(n => (n.id === id ? updatedNote : n)))
      })
      .catch(() => {
        alert(`the note '${note.content}' was already deleted from server`)
        setNotes(notes.filter(n => n.id !== id))
      })
  }

  const addNote = (event: React.FormEvent) => {
    event.preventDefault()
    createNote({ content: newNote, important: Math.random() < 0.5 }).then(createdNote => {
      setNotes(notes.concat(createdNote))
      setNewNote('')
    })
  }

  const notesToShow = showAll ? notes : notes.filter(n => n.important)

  return (
    <div>
      <h1>Notes</h1>
      <ul>
        {notesToShow.map(note => (
          <Note key={note.id} note={note} toggleImportance={() => toggleImportanceOf(note.id)} />
        ))}
      </ul>
      <form onSubmit={addNote}>
        <input value={newNote} onChange={e => setNewNote(e.target.value)} placeholder="a new note..." />
        <button type="submit">save</button>
      </form>
      <button onClick={() => setShowAll(!showAll)}>Show {showAll ? 'Important' : 'All'}</button>
    </div>
  )
}

Compare this to the original's App.jsx: every hook, every handler, the controlled-input pattern, the key-per-list-item — all identical in substance. The 'use client' line at the top is the entire structural change needed to make this component interactive inside a Next.js page; everything React already taught you about state and events in part 1 and 2 still applies unmodified once you're inside a Client Component.

Why the boundary sits here and not higher page.tsx stays a plain Server Component — it does one thing, fetch and hand off — rather than being marked 'use client' itself. Pushing the boundary as far down the tree as possible (onto NotesApp, not onto page.tsx) is the pattern the Next.js docs recommend: less of the app opts out of server rendering, and less JavaScript ships to the browser for parts of the page that never needed to be interactive.

4Retire axios and the hardcoded URL

The original service module does two things worth reconsidering now that frontend and backend are one deployable: it depends on axios for what fetch already does natively, and it hardcodes an absolute URL that assumes the backend runs on a separate origin and port.

part2/src/services/notes.js
import axios from 'axios'
const baseUrl = 'http://localhost:3001/notes'

const create = newObject => {
  const request = axios.post(baseUrl, newObject)
  return request.then(response => response.data)
}
lib/notes.ts
export async function createNote(newNote: Omit<Note, 'id'>): Promise<Note> {
  const response = await fetch('/api/notes', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(newNote),
  })
  return response.json()
}

Both changes come from the same underlying fact: this frontend and its backend are now the same Next.js app, served from the same origin. A relative path like /api/notes resolves correctly wherever the app is running — localhost:3000 in development, whatever domain it's deployed to in production — with no environment-specific base URL to configure and no CORS headers to add on the backend, both of which the two-separate-servers setup in part 2/3 needed. And since this file only ever runs in the browser (it's imported exclusively by the 'use client' component), the Web-standard fetch the browser already ships is all it needs — nothing for axios to add.

The full module, typed and axios-free:

lib/notes.ts
export interface Note {
  id: string
  content: string
  important: boolean
}

export async function getAllNotes(): Promise<Note[]> {
  const response = await fetch('/api/notes')
  return response.json()
}

export async function createNote(newNote: Omit<Note, 'id'>): Promise<Note> {
  const response = await fetch('/api/notes', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(newNote),
  })
  return response.json()
}

export async function updateNote(id: string, updated: Note): Promise<Note> {
  const response = await fetch(`/api/notes/${id}`, {
    method: 'PUT',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify(updated),
  })
  return response.json()
}
One real behavior change, not just a rename The original used axios.patch for toggling importance; the method above uses PUT. That's a deliberate call, not an accident — PATCH (partial update) support isn't part of the app/api/notes/[id]/route.ts backend built earlier in this part, only GET and DELETE are. Wiring up importance-toggling end to end — the route handler, the verb, all of it — is exactly what the exercise at the end of this page asks you to defer, so updateNote here is written for when that exists, not called yet.

getAllNotes is kept in this module too, even though page.tsx no longer needs it (it reads lib/notes-store.ts directly, per step 2) — it's there so any future Client Component that needs to re-fetch the full list (after a delete, say) has a typed helper ready, without reintroducing the server-side self-fetch problem, since this one only ever runs in the browser.

5Fix the input bug and clean up

Two small but real fixes, both visible in the NotesApp.tsx listing above already — called out here explicitly since they're easy to port over by accident if you're translating line-by-line instead of re-reading each line:

Fix — placeholder text as initial state 
// original — submits the literal string if the user never types anything
const [newNote, setNewNote] = useState('a new note...')

// fixed — real placeholder UX belongs on the <input>, not in the state
const [newNote, setNewNote] = useState('')
// ...
<input value={newNote} onChange={e => setNewNote(e.target.value)} placeholder="a new note..." />
Fix — delete the narration

The original App.jsx carries roughly 25 lines of commented-out JSX (an earlier, pre-map version of the list) and inline comments explaining what promises, effects, and controlled components are. Both were genuinely useful the first time through — that's exactly why they're comments and not deleted code in the original teaching material. Neither belongs in the version you keep: dead code invites someone to wonder if it's meant to come back, and comments narrating well-known React concepts just go stale as the surrounding code changes. None of it appears in the NotesApp.tsx listing above.

6Split out the Note component

This one barely changes — it's a good example of code that was already in decent shape and just needs types:

part2/src/components/Note.jsx
const Note = ({ note, toggleImportance }) => {
  const label = note.important
    ? 'make not important' : 'make important'

  return (
    <li>
      {note.content}
      <button onClick={toggleImportance}>{label}</button>
    </li>
  )
}

export default Note
app/components/Note.tsx
import type { Note as NoteType } from '../../lib/notes'

export default function Note({
  note,
  toggleImportance,
}: {
  note: NoteType
  toggleImportance: () => void
}) {
  const label = note.important ? 'make not important' : 'make important'

  return (
    <li>
      {note.content}
      <button onClick={toggleImportance}>{label}</button>
    </li>
  )
}
No 'use client' needed here Note.tsx has no state, no hooks, no browser APIs of its own — by the criteria from step 3 it wouldn't need to be a Client Component even on its own. It doesn't need the directive anyway: any component rendered underneath a 'use client' boundary (here, NotesApp) is part of that client tree automatically. The directive only needs to mark the topmost file where the client boundary begins.

7The full picture

Four files, the same count as the original, doing the same job:

FileServer or ClientResponsibility
app/page.tsxServerFetches initial notes, hands them to NotesApp
app/components/NotesApp.tsxClientAll state, all handlers — the interactive heart of the app
app/components/Note.tsx(inherits Client)Presentational: one note, one toggle button
lib/notes.tsClient-only helpersTyped fetch wrappers for mutations, plus the shared Note type
lib/notes-store.tsServer-onlyThe in-memory data itself, shared by page.tsx and the existing app/api/notes route handlers

Nothing here introduces a concept beyond what part 1 and 2 already covered about React itself — state, props, controlled inputs, lists and keys are all unchanged. What changed is entirely about where code runs: less of it needed to run in the browser at all, and the part that does no longer needs to know or care what port the backend lives on.

Make the backend built earlier in this part (app/api/notes/route.ts and app/api/notes/[id]/route.ts) work with the converted notes frontend above.

Do not implement the functionality for toggling a note's importance yet — that needs a PUT or PATCH export on the [id] route that doesn't exist yet, and wiring that up properly (with the right status codes for a missing or malformed id) is what a later exercise in this part covers.
